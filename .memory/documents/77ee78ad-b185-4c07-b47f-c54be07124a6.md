# Plan: Add Default Fuzzy Search for Research and Plans Queries

## Goal

Add default fuzzy search support to query-research and query-plans. The commands should search both summaries and document bodies, rank results by relevance, and keep output concise by showing summaries only.

## Requirements to preserve

- No new required CLI arguments. Fuzzy matching enabled by default.
- Query type filtering strict: query-research returns only RESEARCH, query-plans returns only PLAN.
- Exact substring matches must still be returned and rank ahead of weaker fuzzy matches.
- Body text participates in matching, but output must not print document bodies.
- --include-invalidated behavior unchanged.
- Related suggestions based on final matched direct result IDs.

## Relevant files

Cargo.toml (add nucleo-matcher dependency), src/commands/query_text.rs (main implementation). No changes to cli.rs, main.rs, output.rs, or db.rs.

## Dependency choice

Use nucleo-matcher for fuzzy scoring. It provides high-quality, deterministic fuzzy-finder style matching with good Unicode behavior. Avoid skim (too heavy for non-interactive ranking) and strsim (whole-string similarity is not ideal for fuzzy subsequence matching).

## Implementation outline

### 1. Refactor candidate evaluation in query_text.rs

Replace current substring-contains loop with a scoring flow: load candidates via db::documents_by_type, for each candidate read the body once, score query against summary and body, combine into a SearchHit, retain only candidates with exact match or fuzzy score above threshold, sort by relevance, convert to Vec<DocumentSummary> for output.

SearchHit struct: summary, score, exact_summary, exact_body, original_index.

### 2. Replace document_body_contains with body loading/scoring helpers

Split into a body-reading helper that returns the body string, then scoring can reuse it without reading twice.

### 3. Scoring design

Trim query, compare case-insensitively. Compute: exact_summary (contains), exact_body (contains), fuzzy score for summary, fuzzy score for body. Prefer summary matches over body-only.

final_score = max(summary_fuzzy + SUMMARY_WEIGHT, body_fuzzy). If exact_summary: final_score += EXACT_SUMMARY_BONUS (1,000,000). Else if exact_body: final_score += EXACT_BODY_BONUS (500,000). SUMMARY_MATCH_BONUS = 10,000.

### 4. Thresholding

Always include exact substring matches. Include fuzzy matches only when above a conservative threshold. For queries shorter than 3 chars, only exact substring matches (no fuzzy).

### 5. Sorting

Sort by score desc, then exact summary before exact body before fuzzy-only, then original_index ascending, then document ID as final tie-breaker.

### 6. Output

Continue to call output::print_query_results with None for bodies (bodies not passed for research/plan queries). No separate Fuzzy Matches section.

### 7. Testing

Ask testing subagent to add integration tests covering: fuzzy typo matching, exact-over-fuzzy ordering, closer-match-over-weaker ranking, type filtering despite stronger cross-type matches, body matching while body stays hidden, no results for noise queries, stable order for equal scores.

## Risks and mitigations

- Overly broad fuzzy matches: use threshold, query length guard, no-results test.
- Large document bodies: fuzzy scoring is more expensive; keep implementation simple first, optimize later by scoring summaries first.
- nucleo-matcher API complexity: encapsulate all crate-specific calls in one helper.
- Output regressions: do not change output.rs.
- Test flakiness: include deterministic tie breakers.

## Acceptance criteria

query-research and query-plans return relevant documents for misspelled/fuzzy queries. Exact matches still work and rank above fuzzy-only. Summary and body both contribute. Bodies not printed. Type/invalidation filtering correct. Unrelated fuzzy noise does not return all documents. cargo test passes.
